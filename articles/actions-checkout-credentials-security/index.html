actions/checkout Credentials: Safer GitHub Actions Clones

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

actions/checkout is often treated as harmless setup, but its token, SSH, submodule, and fetch settings decide what later steps can read or write.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for actions/checkout credentials

  • The actions/checkout README says a single commit is fetched by default, and fetch-depth: 0 fetches all history for all branches and tags. [1]
  • The README says the auth token is persisted for authenticated git commands and removed during post-job cleanup; persist-credentials: false opts out. [1]
  • The README says the default token input is ${{ github.token }}. [1]
  • GitHub's workflow syntax reference says unspecified GITHUB_TOKEN permissions become none once any permission is specified. [2]
  • GitHub's secure-use reference warns that compromised third-party actions can access configured secrets and may use GITHUB_TOKEN to write to the repository. [3]

Checkout credentials should match the job's real job

Bottom line: Treat actions/checkout as the step that hands later commands repository access. Keep persisted credentials off for read-only jobs, give the job only contents: read unless it must write, and avoid broad PATs for ordinary clones.

The checkout action's README says the default token input is ${{ github.token }}, which means checkout normally receives the workflow's built-in repository token. [1]

That is convenient, but it also means every script after checkout should be reviewed against the token and credential settings that checkout prepared for the workspace.

Persisted credentials are useful only for later git writes

The checkout README says the auth token is persisted so scripts can run authenticated git commands, and that post-job cleanup removes the token. [1]

The same README documents persist-credentials: false as the opt-out setting. [1]

Clean verdict: Test, lint, build, and scan jobs usually need the source tree, not a reusable git credential. Leave credentials unpersisted unless a later step must push, tag, fetch a private dependency, or update repository state.

The checkout token should be narrowed at the job level

GitHub's workflow syntax reference says jobs.<job_id>.permissions can modify the default GITHUB_TOKEN permissions for a specific job so the workflow allows only the minimum required access. [2]

The same reference says that when any permission is specified, unspecified permissions are set to none. [2]

A checkout-only test job usually starts with permissions: contents: read. A release or documentation update job may need write scopes, but those jobs should be separate from untrusted pull-request work and easier to inspect.

Personal access tokens expand the blast radius

The checkout README notes that ${{ github.token }} is scoped to the current repository and that checking out a different private repository requires a personal access token. [1]

That is a real boundary. A PAT can cross repository lines, so it should live only in workflows that need the second private repository, with environment protection or review where the repository makes that possible.

Avoid this posture: A general pull-request test workflow carries a broad PAT just because one historical step once fetched a private helper repository.

Fetch depth changes what later commands can inspect

The checkout README says only one commit is fetched by default for the triggering ref or SHA, and fetch-depth: 0 fetches all history for all branches and tags. [1]

That default is often safer and faster for ordinary builds. Jobs that compute changelogs, compare merge bases, generate release notes, or inspect tags may need deeper history, but the reason should be visible in the workflow.

Submodules turn checkout into a dependency fetch

The checkout README documents submodules values of true and recursive, and says SSH URLs are converted to HTTPS when an SSH key is not provided. [1]

If submodules are enabled, checkout is no longer just reading one repository. Review which repositories are fetched, whether they are pinned to expected commits, and whether the token or SSH key can reach more code than the job needs.

A fast checkout audit follows the credential path

List every uses: actions/checkout step. For each one, record the token or SSH input, whether credentials persist, the job's permissions block, the fetch depth, submodule mode, and whether later steps run shell commands from pull-request data.

GitHub's secure-use reference warns that a compromised third-party action can access repository secrets and may use GITHUB_TOKEN to write to the repository. [3]
