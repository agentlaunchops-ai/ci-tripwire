GitHub Runner Groups: Secure Self-Hosted CI Access

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

Runner groups help keep powerful self-hosted runners away from repositories and workflows that do not need them.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for runner group access

  • GitHub Docs says runner groups collect sets of runners and create a security boundary around them. [1]
  • GitHub Docs says self-hosted runner group policies can limit access to self-hosted runners added to an organization. [2]
  • GitHub Docs says runner group access can be limited to selected repositories. [2]
  • GitHub Docs says workflow access restrictions require the full workflow path and a branch, tag, or full SHA. [2]
  • GitHub Docs says groups and labels can be combined so a runner must meet both requirements to run a job. [3]

Runner groups turn runner reach into a CI/CD control

Bottom line: Put self-hosted runners into narrow runner groups, grant only the repositories and workflows that need each group, and treat broad default access as an exposure path.

GitHub Docs describes runner groups as an organization-level boundary for controlling access to grouped runners. [1]

That boundary matters because self-hosted runners can sit on machines, networks, and credentials that are more sensitive than ordinary hosted CI capacity.

The default group is a starting point, not the production boundary

GitHub Docs says new runners are automatically assigned to the default group unless otherwise specified, and runners can only be in one group at a time. [1]

For a production runner, that default should trigger a review. A deploy-capable runner belongs in a purpose-named group that makes the intended access obvious before a workflow can route jobs to it.

Clean verdict: A runner group should describe the runner's trust level, not just the operating system or hardware shape.

Repository and workflow policies decide who can reach the runner

GitHub Docs says organizations can use policies to limit access to self-hosted runners that have been added to an organization. [2]

The same GitHub Docs page describes changing which repositories can access a runner group, including selected-repository access. [2]

Use selected repository access for runner groups that can reach deployment networks, signing material, package publishing paths, or internal services. Public or broad organizational access should be reserved for low-trust runners that have no sensitive network path.

Workflow restrictions make group access reviewable

GitHub Docs says that if workflow access is specified for a runner group, the workflow reference must use the full path, including repository owner and name, and must be pinned to a branch, tag, or full SHA. [2]

That rule gives reviewers something concrete to inspect. Instead of asking whether a repository can use a sensitive runner, ask which workflow file can use it and whether that workflow is protected by branch rules, review gates, and narrow token permissions.

Avoid verdict: Do not grant a production runner group to a repository and then rely on every future workflow author to avoid using it.

Labels route jobs, but labels are not an access policy

GitHub Docs says workflows can use YAML to send jobs to a combination of self-hosted runner labels, and a matching runner becomes eligible to run the job. [3]

The workflow documentation also explains that a job can require both a group and matching labels before a runner is eligible. [3]

Use labels for capability matching, such as architecture, operating system, or special hardware. Use groups for access. A label named prod is not enough if the runner group itself is broadly available.

A fast runner-group audit follows the reachable workflow

Start with every workflow that uses runs-on: self-hosted or a named runner group. Confirm the runner group, selected repository policy, workflow restrictions, branch or tag reference, and label set. Then compare those workflows against the runner's real machine access: local files, cloud credentials, package registries, deployment networks, and long-lived services.

GitHub's secure-use reference warns that an individual attacker can compromise a repository when workflows execute untrusted code on a self-hosted runner, so self-hosted runner access should be reviewed before untrusted pull-request or branch paths can reach it. [4]
