GitHub Actions pull_request: Safer PR CI Workflows

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

pull_request workflows are the ordinary CI path for proposed changes, so their security review should focus on untrusted code, fork behavior, token scope, secrets, artifacts, and shell steps.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for pull_request workflow security

  • GitHub Docs says pull_request workflows run when pull-request activity occurs and can be filtered by activity types, branches, and paths. [1]
  • GitHub Docs says workflows do not run on forked repositories by default and must be enabled in the fork's Actions tab. [1]
  • GitHub Docs says a public repository may require approval before workflow runs from outside collaborators execute. [2]
  • GitHub Docs says workflows should grant GITHUB_TOKEN the least required access with the permissions key. [3]
  • GitHub's secure-use reference says some contexts should be treated as untrusted input because an attacker can insert malicious content. [4]

pull_request CI still runs around untrusted changes

Bottom line: Use pull_request for unprivileged PR checks, but do not treat it as risk-free. Constrain when it runs, who can start forked runs, what token access jobs receive, and how workflow steps handle PR-controlled data.

GitHub Docs says the pull_request event runs workflows when activity on a pull request happens. [1]

The same event reference says teams can use types, branch filters, path filters, and combinations of branch and path filters to narrow when pull_request workflows run. [1]

Trigger filters define which PRs get CI attention

GitHub Docs says pull_request workflows can run for specific activity types such as opened, synchronize, and reopened. [1]

GitHub Docs says branch filters for pull_request evaluate the base branch of the pull request. [1]

GitHub Docs also says path filters can include or exclude changed file paths, and that workflows only run when both branch and path filters match if both are configured. [1]

Clean verdict: A PR workflow is easier to reason about when its activity types, base branches, and paths match the repository areas the job actually protects.

Forked pull requests need an explicit run posture

GitHub Docs says workflows do not run on forked repositories by default and must be enabled in the fork's Actions tab. [1]

GitHub's fork-approval guide says public repositories can require approval before workflow runs from first-time contributors, all outside collaborators, or other configured outside-contributor groups execute. [2]

The same guide says a maintainer with write access can approve or reject a workflow run from a public fork. [2]

That approval step is not only a queue-management choice. It is a security boundary around running CI that may build, test, or package code submitted by someone outside the trusted maintainer set.

PR jobs should start from narrow token permissions

GitHub Docs says the permissions key can modify GITHUB_TOKEN permissions for the entire workflow or for individual jobs. [3]

GitHub Docs says teams should grant the GITHUB_TOKEN the least required access. [3]

GitHub Docs also says an action can access the GITHUB_TOKEN through the github.token context even if the workflow does not explicitly pass it as an input. [3]

Avoid verdict: Do not let a broad default token ride along in a PR workflow just because the workflow appears to be a simple test job.

PR-controlled data should not flow straight into shell

GitHub's secure-use reference says certain contexts should be treated as untrusted input because an attacker could insert malicious content. [4]

The secure-use reference shows a script-injection pattern where pull-request title data is used directly inside a shell script. [4]

The same reference recommends an intermediate environment variable for handling potentially untrusted expression values in shell scripts. [4]

That matters for pull_request workflows because a contributor can influence fields such as titles, branch names, commit data, file paths, and changed content that workflow steps may later consume.

A fast audit treats pull_request as untrusted CI

Start with the trigger. Check whether types, base-branch filters, and path filters match the jobs the workflow performs, because GitHub documents each of those narrowing controls for pull_request. [1]

Then review fork handling. Confirm whether outside-collaborator runs need approval, and make sure maintainers understand that approving a run allows CI to execute submitted workflow-relevant changes. [2]

Next, shrink credentials. Prefer job-level permissions, remove unused secrets, and avoid token access in jobs that only lint, test, or build untrusted code. GitHub's token guide specifically recommends least required access. [3]

Finally, trace PR-controlled data. Move titles, branch names, labels, paths, and other pull-request context values through safer handling before shell execution, especially when a job runs code from an outside contributor. [4]
