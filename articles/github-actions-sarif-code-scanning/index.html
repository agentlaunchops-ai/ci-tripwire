SARIF Uploads: Secure GitHub Actions Code Scanning

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

SARIF uploads turn third-party scanner output into GitHub code scanning alerts, so the workflow that uploads results becomes part of the security boundary.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for SARIF upload workflows

  • GitHub Docs says SARIF files generated outside GitHub can be uploaded so third-party tool findings appear as code scanning alerts. [1]
  • GitHub Docs says the upload-sarif action uses the sarif_file input for the file or directory and an optional category input for result grouping. [1]
  • GitHub's example SARIF upload workflows grant security-events: write to the upload job. [1]
  • GitHub Docs says unspecified token permissions become none after any permission is specified. [4]
  • GitHub Docs says code scanning pull-request alerts can become a required check depending on branch protection rules. [3]

SARIF uploads are a publishing path for security findings

Bottom line: Treat SARIF upload jobs as trusted reporting jobs: give them security-events: write only where needed, keep categories stable, preserve fingerprints, and make pull-request checks enforce the scanner signal you actually trust.

GitHub Docs says third-party tools and CI/CD systems can generate SARIF and upload it to GitHub so findings appear in code scanning. [1]

That makes SARIF upload useful for tools that are not CodeQL. It also means a workflow can affect which alerts appear, how they group, and whether teams see clean pull-request feedback.

The upload job needs security-events, not broad write-all

GitHub's SARIF upload example sets permissions: security-events: write before calling github/codeql-action/upload-sarif@v4. [1]

GitHub's workflow syntax reference says the security-events permission works with code scanning alerts, and that security-events: write allows an action to update the status of a code scanning alert. [4]

Keep that permission on the analysis or upload job instead of using write-all. When any permission is specified, GitHub Docs says all unspecified permissions are set to none, which gives reviewers a compact permission surface to inspect. [4]

Stable categories keep multiple scanner runs separate

GitHub Docs says code scanning expects one SARIF results file per analysis by default, and that a second SARIF file for a commit can replace the original set unless each set is identified as unique. [1]

For GitHub Actions uploads from third-party static analysis tools, GitHub Docs says to specify a category input when more than one file of results for the same tool is uploaded in one workflow. [1]

Clean verdict: Name categories after a stable analysis slice, such as gha-guard/workflows, not after a temporary matrix value or branch name.

Fingerprints decide whether alerts duplicate or track cleanly

GitHub Docs says it uses the SARIF partialFingerprints property to detect when two results are logically identical. [2]

The SARIF support guide says the upload-sarif action will create partialFingerprints when they are missing, but only when the repository contains both the SARIF file and the source code used in the static analysis. [1]

The same support guide warns that API uploads without fingerprint data are processed and displayed, but users may see duplicate alerts. [2]

Pull-request checks need the same scanner signal every time

GitHub Docs says code scanning alerts can appear on pull requests as check results and annotations when code scanning is configured as a pull-request check or when push results are mapped to open pull requests. [3]

GitHub Docs also says the Code scanning results check may be a required check that prevents merges, depending on branch protection rules. [3]

Do not make that required check depend on a workflow that changes scanner categories, skips uploads silently, or runs with unnecessary repository write access. A required check is only useful if the result means the same thing on every protected branch.

A fast SARIF upload audit starts with the workflow file

Review each workflow that uploads SARIF. Confirm the trigger, scanner command, generated file path, upload action version, security-events permission, category strategy, fingerprint behavior, and whether the check is required by branch protection.

GitHub Docs says SARIF results need at least one location for code scanning to display a result, and recommends relative repository paths for artifact URIs. [2]
