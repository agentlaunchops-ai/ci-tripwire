GitHub Actions continue-on-error: Keep CI Failures Visible

By CI Tripwire Editorial · Published 2026-06-09 · 7 min read

GitHub Actions continue-on-error is useful for experimental checks, but it should be easy to explain before a failing job or step stops blocking the workflow.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for GitHub Actions continue-on-error

  • GitHub Docs documents jobs.<job_id>.steps[*].continue-on-error as a way to prevent a job from failing when a step fails. [1]
  • GitHub Docs documents jobs.<job_id>.continue-on-error as a way to prevent a workflow run from failing when a job fails. [1]
  • GitHub Docs says jobs.<job_id>.strategy.fail-fast applies to the whole matrix and cancels queued or in-progress matrix jobs when a matrix job fails. [1]
  • GitHub Docs says jobs run in parallel by default and that needs identifies jobs that must complete successfully before a dependent job runs. [2]
  • GitHub's secure-use reference warns that some workflow contexts contain untrusted input and recommends careful handling in scripts. [3]

continue-on-error should make risk visible, not invisible

Bottom line: Use continue-on-error for non-blocking experiments, compatibility probes, and soft telemetry. Avoid it on tests, scans, releases, and deploy jobs whose failure should stop merge or delivery.

GitHub Docs describes continue-on-error at both step and job scope. A step-level setting prevents a job from failing when that step fails, while a job-level setting prevents a workflow run from failing when that job fails. [1]

That difference is the review hinge. A tolerated failure can be harmless when a nightly probe tests a future runtime. It can be dangerous when the same setting hides a broken test, skipped scanner, failed package publish, or deployment guard.

Step-level and job-level tolerance change different signals

GitHub Docs says jobs.<job_id>.steps[*].continue-on-error prevents a job from failing when a step fails. [1]

That setting belongs under a specific step, so the local question is narrow: why may this command fail, and what later step records that failure visibly?

GitHub Docs separately says jobs.<job_id>.continue-on-error prevents a workflow run from failing when a job fails. [1]

A job-level exception deserves a stronger reason because it changes the run's overall failure signal. If the job is named release, deploy, security, test, scan, or attest, reviewers should assume the exception is suspicious until the workflow explains it.

Clean verdict: A named experimental matrix leg with continue-on-error is easier to defend than a production deploy job whose failure no longer fails the workflow.

Required checks only help when the failing signal stays blocking

GitHub Docs says jobs run in parallel by default, and it says jobs.<job_id>.needs identifies jobs that must complete successfully before a dependent job runs. [2]

That success chain matters when branch protection or rulesets depend on status checks. A required check that no longer fails when its meaningful command fails can give reviewers a green merge path while the important work is broken.

Matrix jobs need the same review. GitHub Docs says strategy.fail-fast applies to the whole matrix and cancels queued or in-progress jobs when a matrix job fails. [1]

GitHub Docs also says other matrix jobs continue when a job marked continue-on-error: true fails. [1]

That can be useful for an experimental runtime, but the matrix value should make the exception obvious. A reviewer should not need to infer from a long matrix include block that a failed release-critical platform is tolerated.

Tolerated failures are risky when workflows handle untrusted input

GitHub's secure-use reference warns that some contexts contain untrusted input and should be treated carefully in scripts. [3]

That warning becomes more important when a workflow also tolerates failure. A shell step that parses pull-request titles, issue text, branch names, dispatch inputs, or release notes should fail closed when validation fails.

Do not let continue-on-error turn a validation or scanner step into a warning that nobody sees. If a later job publishes an artifact, writes a package, deploys infrastructure, uploads SARIF, comments on pull requests, or creates a release, the earlier guardrail should either block that path or write a clearly reviewed status.

Avoid verdict: A failed validation step should not be tolerated if later jobs still receive tokens, secrets, deployment authority, or release permissions.

A fast audit starts by finding every tolerated failure

Search workflow files for continue-on-error. For each match, record whether it sits on a step or a job, because GitHub Docs gives those placements different failure effects. [1]

Then classify the job. Tolerated failures are usually reasonable for experiments, optional reports, flaky external probes, and compatibility checks. They are harder to justify for required tests, scanners, release jobs, deployment gates, and anything with write permissions.

Finally, trace the downstream path. If a tolerated step or job feeds needs, artifacts, outputs, cache state, release notes, deployment decisions, or package publishing, reviewers should confirm the workflow still fails closed where it matters.
