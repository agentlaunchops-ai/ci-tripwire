GITHUB_TOKEN Permissions: Least Privilege for Actions

By CI Tripwire Editorial · Published 2026-06-07 · 7 min read

Start every workflow with explicit `GITHUB_TOKEN` permissions, then grant write access only to the jobs that need it. That makes privilege visible in code review instead of hiding it in repository defaults.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for `GITHUB_TOKEN` scopes

  • GitHub says `permissions` can modify default `GITHUB_TOKEN` access at the workflow level or job level so only required access is allowed. [1]
  • GitHub documents `read`, `write`, and `none` as per-scope permission values; `write` includes `read`. [1]
  • Once any permission is specified, GitHub sets permissions that are not specified to `none`. [1]
  • GitHub warns that an action can access `GITHUB_TOKEN` through the `github.token` context even when the workflow does not explicitly pass it. [2]
  • OpenSSF Scorecard gives its highest Token-Permissions result when workflows set read-only permissions at the top level and declare required write permissions at the job level. [4]

Make workflow privilege explicit before adding exceptions

Bottom line: use a narrow top-level baseline such as `permissions: read-all` or `permissions: contents: read`, then add write scopes only on the release, package, Pages, issue, or pull-request jobs that truly need them.

GitHub's workflow syntax reference says the `permissions` key can add or remove default access for the `GITHUB_TOKEN`, either for an entire workflow or for a specific job. [1] GitHub's token guide also says workflows should grant the token the least required access. [2]

The payoff is review clarity. A maintainer can scan one file and see which job can write contents, packages, pull requests, Pages, statuses, or security events.

A read-only baseline turns missing scopes into review signals

GitHub documents `read-all`, `write-all`, and `{}` as shorthand for broad read access, broad write access, or disabled permissions. [1] It also documents a sharper behavior: if a workflow specifies access for any permission, every permission not specified is set to `none`. [1]

That behavior makes explicit baselines useful. A workflow that only needs to check out code can start with read access. If a later step needs to publish a release or comment on a pull request, that exception has to appear in the YAML.

  • Good default: top-level `permissions: contents: read` for jobs that only read repository content.
  • Good default: top-level `permissions: read-all` when several read scopes are genuinely useful.
  • Sharper default: top-level `permissions: {}` for workflows where each job should declare every token capability.

Job-level write access should name the reason

GitHub documents `jobs.<job_id>.permissions` for changing the `GITHUB_TOKEN` permissions of a specific job, including giving different jobs different permission sets. [1] That is the right place for most write scopes, because it keeps the blast radius near the action that uses the capability.

Examples are straightforward: `issues: write` for a job that opens an issue, `pull-requests: write` for a job that labels a pull request, `pages: write` for a Pages deployment, `packages: write` for package publishing, or `contents: write` for release creation. GitHub's permission table documents what those scopes allow. [1]

Review rule: a job with `write` should have a narrow job name, a small step list, and no unrelated build, test, dependency install, or untrusted input handling.

Fork pull requests have special token rules

GitHub's workflow syntax reference says forked repositories can add and remove read permissions through the `permissions` key, but typically cannot grant write access. [1] GitHub documents an exception when an admin has enabled the setting that sends write tokens to workflows from pull requests. [1]

The same reference says `pull_request_target` receives read/write repository permission even when triggered from a public fork. [1] That is why token permissions and trigger choice should be reviewed together, not as separate checklist items.

  • Fork PR path: keep ordinary pull-request jobs read-only unless there is a deliberate exception.
  • Privileged PR path: avoid running pull-request code in jobs that can write back to the base repository.

`write-all` is a loud exception, not a convenience setting

GitHub documents `write-all` as a way to grant write access for all available permissions. [1] That breadth is rarely the honest shape of a CI job. A Pages deploy does not need issue write access, a labeling job does not need package write access, and a release job should not quietly inherit unrelated scopes.

GitHub's secure-use reference warns that a compromised action can have significant impact because it may access repository secrets and use `GITHUB_TOKEN` write access. [3] OpenSSF Scorecard's Token-Permissions guidance says required write permissions should be set at the job level and says not to set `permissions: write-all` at the job level. [4]

Caution: when `write-all` appears, treat it as a security exception that needs a written reason and a tighter replacement plan.

A five-minute audit starts with four searches

Search `.github/workflows` for `permissions:`, `write-all`, `pull_request_target`, and `secrets.GITHUB_TOKEN`. Those strings map to the source-backed questions in this guide: where token access is declared, whether broad write access exists, whether a privileged pull-request trigger is present, and whether steps pass the token explicitly. [1] [2]

Then scan for workflows with no `permissions:` block at all. Those files rely on enterprise, organization, or repository defaults before workflow and job adjustments are applied. GitHub documents that permission calculation order in the workflow syntax reference. [1]

For a local review queue, run gha-guard and inspect any token-permissions, trigger, checkout, or shell-context findings before changing release automation.
