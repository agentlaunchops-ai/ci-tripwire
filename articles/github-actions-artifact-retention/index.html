GitHub Actions Artifacts: Retention and Access

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

Workflow artifacts are handy build outputs, but they also become long-lived evidence and sometimes long-lived exposure.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for GitHub Actions artifacts

  • GitHub says artifacts let workflows share data between jobs and store data after a workflow completes. [1]
  • The official upload-artifact action says artifacts are retained for 90 days by default, and retention-days can set a shorter period. [3]
  • The same upload-artifact docs say the retention period must be between 1 and 90 inclusive. [3]
  • GitHub's artifact REST API can list, retrieve, download, and delete workflow artifacts. [2]
  • GitHub Docs say upload-artifact returns a SHA256 digest and download-artifact validates the downloaded artifact against it. [1]

Artifact retention should match the review window

Bottom line: Treat artifacts as published CI outputs. Name them clearly, keep them only as long as the team needs them, and audit who can fetch them after the run.

GitHub describes workflow artifacts as files that can move between jobs and remain available after the workflow run finishes. [1]

That is useful for test reports, build bundles, coverage output, and release candidates. It also means a workflow can preserve generated files after the runner is gone, so retention and download posture deserve the same review as caches, logs, and release assets.

Short retention limits stale build output exposure

The official upload-artifact action documentation puts default artifact retention at 90 days, unless repository settings lower that maximum. It also documents the retention-days input for shorter retention. [3]

The same action documentation allows retention values from 1 through 90 days. [3]

Clean verdict: A pull-request diagnostic bundle might need a few days; a release candidate might need a longer review window; neither should inherit a default without a reason.

For routine CI output, start by asking how long reviewers actually look at the artifact. If the answer is "until the pull request merges," a short explicit retention value is easier to justify than a repository-wide default.

Artifact downloads can cross job and workflow boundaries

GitHub's artifact tutorial documents upload and download actions for sharing data between jobs. It also notes that downloading artifacts from a different workflow or workflow run requires a token and run identifier. [1]

That distinction matters in security reviews. Same-run handoff is a normal workflow pattern; cross-run or cross-repository artifact fetching is closer to consuming an external input and should be reviewed for trust, naming, and token scope.

If a deployment job downloads a build artifact created earlier, reviewers should be able to answer which run produced it, which workflow uploaded it, which branch or ref triggered that run, and whether the deployment job trusts that source.

Upload inputs decide what gets preserved

The upload-artifact README documents inputs including path, retention-days, compression-level, overwrite, and include-hidden-files. [3]

Those inputs are not just convenience settings. A broad path can capture more files than intended. Hidden-file upload can preserve dotfiles. Overwrite behavior can change artifact identity, because the action documentation says overwriting gives the artifact a new ID and removes the previous one. [3]

Avoid this posture: A workflow uploads an entire workspace because it is easy, then depends on humans noticing if secrets, dotfiles, or generated credentials land in the archive.

Digest validation helps detect changed artifact contents

GitHub Docs say each upload-artifact use returns an output named digest, a SHA256 digest of the uploaded artifact. [1]

GitHub Docs also say download-artifact automatically calculates the downloaded artifact digest and validates that it matches the upload step's digest; if the digest does not match, the run displays a warning in the UI and job logs. [1]

Use that as a review signal, not as a complete release-provenance system. For release builds, pair artifact handling with workflow security controls such as pinned third-party actions and restricted token permissions. [4]

The REST API makes stale artifacts auditable

GitHub's REST API documentation says the Actions artifacts API can download, delete, and retrieve information about workflow artifacts. [2]

The documented artifact object includes fields such as expired, created_at, expires_at, and archive_download_url. [2]

That gives maintainers a practical cleanup path: list artifacts by repository or workflow run, flag old artifacts with broad names, compare expiration against policy, and delete artifacts that no longer support debugging, compliance, or release review.

A fast artifact audit starts with upload steps

Search workflow files for actions/upload-artifact and record the artifact name, path, retention setting, hidden-file posture, overwrite setting, and downstream download steps. Then check whether deployment jobs download artifacts from the same workflow run or from another run or repository.

Next, compare the artifact list from the REST API with the workflow inventory. Unexpected names, long expiration windows, and cross-run downloads deserve review before the artifacts become part of release automation.
