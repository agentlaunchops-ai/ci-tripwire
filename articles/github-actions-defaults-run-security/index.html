GitHub Actions defaults.run: Safer CI Shell Controls

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

GitHub Actions defaults.run looks like workflow cleanup, but it can silently change the shell and working directory used by many script steps.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for defaults.run security

  • GitHub Docs says defaults.run can provide default shell and working-directory options for every run step in a workflow. [1]
  • GitHub Docs says job-level defaults override workflow-level defaults when more than one default setting uses the same name. [1]
  • GitHub Docs says contexts and expressions are not allowed in defaults.run. [1]
  • GitHub Docs warns that some contexts contain untrusted input and that attackers could insert malicious content. [3]
  • GitHub Docs says the default working directory is GITHUB_WORKSPACE, which is the default directory on the runner for steps and the checkout location when actions/checkout is used. [4]

defaults.run is executable policy, not cosmetic YAML

Bottom line: Review defaults.run as a workflow-wide script policy. A shell or directory default can change how every later command expands variables, finds files, and fails.

GitHub Docs says defaults.run can set default shell and working-directory values for every run step in a workflow. [1]

That makes the block useful for reducing repeated YAML, but it also makes it easy for reviewers to miss behavior that applies far below the place where it is declared.

Workflow and job defaults can disagree

GitHub Docs says a workflow can use defaults to create a map of default settings that apply to all jobs in the workflow. [1]

The same syntax reference says a job can also set defaults, and when more than one default setting is defined with the same name, GitHub uses the most specific default. [1]

In practice, reviewers should check both levels. A workflow-level shell may look strict, while a specific deploy job changes the shell or directory for the highest-risk commands.

Clean verdict: Treat every job-level defaults.run override as a meaningful change to the review surface.

Shell choice changes quoting and failure behavior

GitHub Docs lists supported shell values for run steps, including platform-specific options such as bash, pwsh, python, sh, cmd, and powershell. [1]

The workflow syntax reference also documents that the unspecified shell on Linux and macOS runs a different internal command than explicitly specifying bash. [1]

Those details matter when a workflow receives branch names, issue text, pull-request titles, dispatch inputs, or generated filenames. GitHub Docs warns that some contexts should be treated as untrusted input because attackers could insert malicious content. [3]

GitHub's secure-use reference says the preferred approach for inline scripts is to set expression values to an intermediate environment variable before the script uses them. [2]

Working-directory defaults decide which files commands touch

GitHub Docs says defaults.run.working-directory provides the default working directory for all run steps in a workflow. [1]

The variables reference says GITHUB_WORKSPACE is the default working directory on the runner for steps and the default location of the repository when using actions/checkout. [4]

A changed working directory can make scripts read a different package manifest, use a different lockfile, write outputs into a sibling project, or miss the intended repository path. That is not only a build concern; it is also a review concern for release, scan, and deployment jobs.

Avoid verdict: Do not approve a directory default without checking the commands that inherit it.

A fast audit starts at inherited run behavior

First, search each workflow for defaults:, then inspect both workflow-level and job-level defaults.run blocks. GitHub Docs says job defaults can override workflow defaults, so the local job setting wins when names collide. [1]

Second, compare the declared shell with the commands below it. Treat shell changes as security-relevant when commands include contexts, filenames, environment variables, or generated output.

Third, trace working-directory defaults into release, publish, scan, and deploy steps. The default workspace is predictable, but an override can move commands away from the checkout path that reviewers expect. [4]
