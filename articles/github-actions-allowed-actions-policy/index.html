GitHub Actions Allowlist: Control Third-Party Actions

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

An Actions allowlist is useful when it turns third-party automation from a default permission into an explicit dependency decision.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for GitHub Actions allowed-actions policy

  • Repository settings can limit GitHub Actions to all actions, local-only actions, or selected actions and reusable workflows. [1]
  • GitHub says selected-action entries can target an action by owner, repository, tag, or commit SHA, and can use wildcard patterns. [1]
  • Organization owners can enable, disable, and limit GitHub Actions for an organization, including which repositories may run Actions. [2]
  • The Actions permissions REST API exposes repository policy fields including allowed_actions and sha_pinning_required. [3]
  • GitHub's secure-use reference recommends pinning third-party actions to a full-length commit SHA when workflows depend on external actions. [4]

Allowed-actions policy makes external automation reviewable

Bottom line: Start with selected actions, keep local and trusted internal workflows easy to use, and make every outside action or reusable workflow earn a place on the allowlist.

GitHub lets repository administrators manage which actions and reusable workflows can run in a repository. The documented options include allowing all actions, allowing only local actions and reusable workflows, or allowing selected actions and reusable workflows. [1]

That setting is a supply-chain control because a workflow dependency can execute code during CI. The useful posture is not "block everything forever"; it is "new automation needs a review path before it runs with repository context, tokens, caches, artifacts, or deployment credentials."

Selected actions create an explicit dependency boundary

When selected actions are used at the repository level, GitHub documents separate choices for GitHub-created actions, Marketplace actions by verified creators, and specified actions or reusable workflows. [1]

This helps reviewers separate three cases: first-party actions such as common GitHub-owned workflow building blocks, trusted organization-owned workflows, and outside dependencies that need owner, version, and permission review.

Clean verdict: A repository can still run approved automation while unknown third-party actions fail policy instead of quietly becoming part of CI.

Organization policy prevents repository drift

GitHub documents organization-level controls for enabling, disabling, and limiting GitHub Actions, including policies that restrict which repositories can use Actions. [2]

That matters when repository administrators make local changes under deadline pressure. If the organization owns the baseline, teams can grant exceptions through a deliberate process instead of discovering later that one repository allowed all actions while another used selected actions.

GitHub also notes that workflow approval policies are intended to restrict users who can execute workflows that could cause unexpected resource and compute consumption, and that self-hosted runners add risk if user-controlled workflow code is allowed to execute. [2]

Allow patterns should be narrow enough to explain

GitHub says specified actions cannot exceed 1000 entries, and it documents the syntax for targeting an action by OWNER/REPOSITORY@TAG-OR-SHA. It also documents wildcard matching and a ! prefix for blocking patterns. [1]

Use patterns as policy language, not as a way to recreate "allow all" with extra punctuation. A short list of named repositories and pinned versions is easier to review than broad owner wildcards that include tools nobody has evaluated.

Avoid this posture: A selected-actions policy allows a broad wildcard for a vendor namespace without recording which actions, versions, and permissions the workflows actually use.

Full SHA pinning narrows version-change risk

GitHub's repository settings documentation says the "Require actions to be pinned to a full-length commit SHA" setting requires all actions, including organization-owned and GitHub-authored actions, to be pinned to a full-length commit SHA; reusable workflows can still be referenced by tag. [1]

The secure-use reference also recommends pinning third-party actions to a full-length commit SHA because it is the only way to use an action as an immutable release. [4]

Pinning does not prove an action is safe. It makes the reviewed code identity stable. Pair it with an allowlist, dependency review, permission review, and a routine update process so old SHAs do not become abandoned trust anchors.

The REST API makes policy drift auditable

GitHub's Actions permissions REST API documents repository policy fields for whether Actions is enabled, whether allowed actions are all, local_only, or selected, and whether SHA pinning is required. [3]

The same REST documentation includes selected-action endpoints where policy can represent whether GitHub-owned actions are allowed, whether verified Marketplace actions are allowed, and which action patterns are permitted. [3]

Those endpoints are useful for periodic checks across many repositories. A policy review can flag repositories that reverted to all actions, selected-action lists that grew too broad, or repos where SHA pinning is not enabled.

A fast allowlist audit starts with every uses line

Search workflow files for uses:, then sort each dependency into local action, reusable workflow, GitHub-owned action, verified Marketplace action, and other third-party action. For each outside dependency, record the business purpose, repository owner, pinning style, token permissions, and whether the action touches secrets or deployment jobs.

Next, compare that inventory with repository and organization policy. Selected actions should match actual workflow needs, broad wildcards should have an owner, and full SHA pins should be expected for third-party actions where the secure-use recommendation applies.

Run gha-guard locally so allowlist review sits beside pinned-action, token-permission, timeout, pull-request trigger, OIDC, reusable-workflow, runner, cache, attestation, environment, dependency-review, shell-context, required-check, and script-injection audits.
