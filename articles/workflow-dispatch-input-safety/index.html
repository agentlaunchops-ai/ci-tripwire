workflow_dispatch Inputs: Safer Manual GitHub Actions

By CI Tripwire Editorial · Published 2026-06-07 · 7 min read

A manual workflow is not automatically safer because a human clicks the run button. Treat `workflow_dispatch` inputs as a small control surface that can still select targets, modes, and shell values.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for workflow_dispatch input safety

  • GitHub documents `workflow_dispatch` as a manual trigger that can define inputs passed to the workflow, and it only receives events when the workflow file is on the default branch. [1]
  • `workflow_dispatch` inputs support `boolean`, `choice`, `number`, `environment`, and `string` types; GitHub also documents a 25-property limit and a 65,535-character input payload limit. [1]
  • The `inputs` context is available to jobs and steps in manually triggered workflows, and GitHub says its values can be strings, numbers, booleans, or choices. [2]
  • GitHub warns that direct expression expansion inside shell scripts can create script-injection risk, and its hardening guide recommends routing untrusted values through intermediate environment variables. [3] [4]
  • Environment protection rules can require reviewers before a job proceeds, and environment secrets are only available after configured rules pass. [5]

Manual workflow inputs should be constrained first

Bottom line: Use `choice`, `boolean`, and `environment` inputs where possible, keep free-form strings away from inline shell expansion, and put production deploys behind environment protection.

Manual runs are common for releases, hotfix deploys, database tasks, and rollback jobs. That makes the input form part of the release control plane. GitHub's syntax reference says `workflow_dispatch` can define inputs passed to a workflow, and that those events only arrive when the workflow file is on the default branch. [1]

The security goal is not to remove manual workflows. The goal is to make each manual decision narrow, typed, reviewable, and hard to accidentally route into a privileged shell command.

A run button can still launch a privileged workflow

A manually triggered workflow can publish a package, deploy to an environment, write to the repository, or reach cloud credentials if the job is configured that way. The click is only the trigger. The permissions, secrets, environment, and shell steps still decide the blast radius.

Control surface: Review `workflow_dispatch` inputs beside `permissions`, `environment`, secret use, and every `run:` block.

GitHub's inputs context reference describes the `inputs` object as available in reusable workflows and in workflows triggered by `workflow_dispatch`, and says it can be accessed from any job or step. [2] That is useful, but it also means a single input can influence conditions, commands, action arguments, deployment targets, and job names.

Typed inputs reduce free-form release choices

GitHub documents five `workflow_dispatch` input types: `boolean`, `choice`, `number`, `environment`, and `string`. [1] Prefer finite choices for deploy targets and release modes, booleans for toggles, and environment inputs where the workflow should connect a selection to GitHub environment controls.

  • Good fit for `choice`: `staging`, `canary`, or `production` target names with a fixed option list.
  • Good fit for `boolean`: a dry-run toggle that gates publish or deploy steps.
  • Good fit for `environment`: a manual deploy that should bind to environment protection and environment-scoped secrets.
  • Use `string` carefully: branch names, image tags, ticket IDs, and migration names need validation before they reach commands.

The syntax reference also documents operational limits: a maximum of 25 top-level input properties and a maximum input payload of 65,535 characters. [1] Those limits are not validation rules. They only bound the shape of the payload GitHub accepts.

Inline shell expansion is the risky input boundary

GitHub's script-injection documentation explains that workflow scripts can become vulnerable when untrusted values are evaluated inside generated shell code. [3] Manual inputs should be treated with the same care as issue titles, branch names, and pull-request metadata when they are interpolated into `run:` steps.

Avoid this pattern: `run: deploy --target "${{ inputs.deploy_target }}"` gives expression expansion a direct path into shell text.

GitHub's hardening guide recommends using an intermediate environment variable for untrusted input in inline scripts. [4] The safer pattern is to assign the expression under `env:`, then reference the native shell variable inside the script. That keeps the workflow expression engine from splicing the input directly into the script body.

Production choices need environment gates

For deploy workflows, a typed input should not be the only guard. GitHub documents environment protection rules with required reviewers, including an option to prevent users from approving workflow runs they triggered. [5]

GitHub also says jobs that use an environment can access environment secrets only after configured rules, such as required reviewers, pass. [5] That makes the environment boundary a stronger place for production credentials than a broad repository secret used by every manual run.

A fast audit starts with every manual trigger

Search `.github/workflows` for `workflow_dispatch`, then review each input name, type, default, and every use of `inputs.` or `github.event.inputs`. Check whether free-form strings reach `run:` blocks, whether deploy targets use finite choices, whether production jobs use environment protection, and whether the job has narrower permissions than the repository default. [1] [2] [5]

For a local review queue, run gha-guard and inspect manual-trigger findings beside token-permission, pinned-action, timeout, and shell-context findings.
