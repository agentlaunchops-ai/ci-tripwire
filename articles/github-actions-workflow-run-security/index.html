GitHub Actions workflow_run: Safe Privileged Follow-ups

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

The workflow_run trigger can be the right way to separate untrusted CI from a privileged follow-up, but only if the handoff is narrow, verified, and explicit.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for workflow_run security

  • GitHub documents workflow_run activity types of completed, requested, and in_progress. [1]
  • GitHub says a workflow_run workflow can access secrets and write tokens even when the previous workflow could not. [1]
  • GitHub warns that running untrusted code on workflow_run can lead to cache poisoning and unintended access to write privileges or secrets. [1]
  • GitHub says workflow_run chains cannot exceed three levels. [1]
  • GitHub Security Lab guidance describes the intended split: untrusted pull-request work runs without secrets, then a privileged workflow_run job processes limited artifacts. [5]

workflow_run is a privilege handoff, not a retry button

Bottom line: Use workflow_run when an unprivileged workflow needs a separate privileged follow-up. Do not let that follow-up rebuild untrusted code, trust arbitrary artifacts, or inherit broad write permissions by default.

GitHub's workflow event reference says workflow_run can run when another workflow run is requested or completed. [1]

The useful part is also the dangerous part: GitHub says the triggered workflow can access secrets and write tokens even if the earlier workflow could not. [1]

The first workflow should handle untrusted work

GitHub Security Lab describes a safe pattern where the untrusted pull-request workflow runs under pull_request, stores limited results as artifacts, and exits without secrets or repository write authority. [5]

The privileged workflow_run job then reads those results and performs the trusted action, such as commenting on a pull request or calling a third-party service that needs a token. [5]

Clean verdict: Keep builds, tests, package installation, and attacker-controlled scripts in the unprivileged workflow. Keep comments, deployments, release writes, and secret-backed calls in the follow-up.

workflow_run filters should name the expected source

GitHub's syntax reference documents on.workflow_run.branches and on.workflow_run.branches-ignore filters for narrowing which triggering workflow branches start a follow-up. [3]

The event reference also says that if multiple workflows are listed for a workflow_run trigger, only one of those workflows needs to run for the follow-up to start. [1]

That makes explicit naming important. A privileged follow-up should identify the upstream workflow it expects, the branch posture it accepts, and the conclusion it requires before any write or secret-backed step runs.

Artifacts from untrusted CI are inputs, not proof

GitHub warns in the event reference that untrusted code on workflow_run can create security vulnerabilities including cache poisoning and unintended access to secrets or write privileges. [1]

GitHub Security Lab's pattern passes only the pull request number and result files through artifacts before the privileged workflow comments back. [5]

That is the right mental model: an artifact is a message from a less-trusted workflow. Parse it as data, avoid executing it, validate the fields you expect, and do not let it choose a shell command, release ref, deployment target, or token scope.

Avoid this posture: A privileged workflow_run job downloads an artifact from a pull-request build and then runs a script, package manager, or deploy command from that artifact.

Token permissions still need an explicit block

GitHub's workflow syntax reference says permissions can define access for the GITHUB_TOKEN and that unspecified permissions are set to none when any permission is specified. [3]

The same reference lists sensitive scopes such as contents: write for creating releases, deployments: write for deployments, packages: write for package publishing, and id-token: write for fetching an OIDC token. [3]

A workflow_run job that only comments should not inherit deployment, package, Pages, attestation, or OIDC authority. Start with the smallest job-level block and expand one permission at a time when the step proves it needs that scope.

workflow_run context data still crosses trust boundaries

GitHub's contexts reference says the github context contains workflow-run and trigger information, and warns that some contexts should be treated as untrusted input because attackers can insert malicious content. [4]

That warning applies to follow-up workflows too. Branch names, pull-request titles, artifact names, commit messages, and event payload fields should be passed through environment variables, quoted carefully, or parsed as structured data before they reach shell commands.

A fast workflow_run audit starts with the follow-up job

List every workflow with on: workflow_run. For each one, record the upstream workflow names, accepted branches, accepted conclusions, requested token permissions, secrets used, artifacts downloaded, and whether the job can comment, merge, deploy, publish, or mint an OIDC token.

Then review whether the privileged job ever checks out untrusted code, executes downloaded artifacts, consumes caches from an untrusted build, interpolates event data into shell commands, or lets the upstream workflow choose a deployment target.
