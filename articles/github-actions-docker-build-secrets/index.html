GitHub Actions Docker Build Secrets: Safer Container CI

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

Docker build secrets need a different review path from ordinary workflow secrets because the sensitive value crosses from the runner into a build container.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for Docker build secrets in GitHub Actions

  • Docker Docs describes build secrets as sensitive values, including passwords or API tokens, that a build consumes. [1]
  • Docker Docs says build arguments and environment variables are inappropriate for passing secrets to builds. [1]
  • Docker Docs says secret mounts, SSH mounts, and Git authentication for remote contexts are the supported build-secret types. [1]
  • Docker Docs maps docker/build-push-action secret inputs to inline values, runner environment variables, and runner files. [2]
  • GitHub Docs says secrets are not passed to workflows triggered from forks, except for GITHUB_TOKEN. [3]

Docker build secrets belong in the CI trust review

Bottom line: Review every Docker build secret as a transfer from GitHub Actions into BuildKit. The workflow, the Dockerfile, and the build context all need to agree that the value is mounted briefly, read intentionally, and not copied into layers, logs, outputs, or artifacts.

In Docker's build documentation, the secret category covers sensitive build-time values such as passwords and API tokens. [1]

That makes build secrets a CI boundary, not only a Dockerfile detail. In GitHub Actions, the value often begins as a GitHub secret, a runner environment variable, an SSH credential, or a file created during the job before it reaches the build step.

Build args and environment variables are the wrong secret channel

For secret material, Docker's build guidance points reviewers away from ordinary build arguments and runner environment values, and toward purpose-built secret or SSH mounts. [1]

The review question is simple: if a value grants registry, package, cloud, or source access, it should not travel as a normal build argument. A reviewer should expect a dedicated secret mount, a dedicated SSH mount, or a remote-context authentication secret.

Avoid verdict: Do not put tokens, package registry credentials, cloud keys, or private source credentials in Docker build args just because the workflow already has them as GitHub secrets.

Build-push-action secret inputs have different source risks

Docker Docs says docker/build-push-action supports secret mounts in GitHub Actions and shows examples using GitHub secrets with the action. [2]

Docker Docs maps secrets to inline workflow values, secret-envs to runner environment variables, and secret-files to files on the runner filesystem. [2]

Those three sources deserve separate review. Inline values are easy to spot in YAML. Environment variables may be set several steps earlier. Secret files can be assembled from templates, downloads, or previous commands, so reviewers should trace the file creation path before trusting the build input.

Dockerfiles should read mounted secrets without persisting them

Docker Docs says secret mounts expose secrets to build containers as files or environment variables. [1]

Docker Docs describes using a secret mount by passing the secret through the build command and then consuming it in the Dockerfile with RUN --mount=type=secret. [1]

The Dockerfile review should follow the value after it is mounted. A package install, private source fetch, or license check can read a mounted secret; a later COPY, cache directory, build output, or diagnostic command should not preserve it.

Clean verdict: The best Docker build-secret path is visible in the workflow, visible in the Dockerfile, and absent from image layers, summaries, uploaded artifacts, and logs.

Fork and pull-request builds need a separate secret posture

GitHub's Actions secret guidance sets a fork boundary: fork-triggered workflows do not receive repository secrets, with GITHUB_TOKEN called out as the exception. [3]

That default helps, but reviewers still need to check how the workflow is triggered. A Docker build running on untrusted pull-request code should not receive registry publish credentials, private package tokens, or cloud keys through a build mount.

GitHub Docs recommends intermediate environment variables when inline scripts handle untrusted input. [4]

The same posture helps before Docker builds. Treat branch names, PR titles, generated Docker tags, build arguments, and file paths as data until the workflow proves they cannot change which secret is mounted or where a built image is pushed.

A fast Docker build-secret audit traces each credential twice

First, list each docker/build-push-action step and classify every secrets, secret-envs, secret-files, and SSH input. Mark whether the source is a GitHub secret, a runner environment variable, a generated file, or a persistent credential.

Second, open the Dockerfile and trace every RUN --mount=type=secret or SSH mount consumer. The build should need that secret for a specific fetch or install, and the command should avoid printing, copying, caching, or exporting the value.
