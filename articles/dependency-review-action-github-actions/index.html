Dependency Review Action: Secure GitHub Actions PRs

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

Dependency review is most useful when it fails pull requests for the dependency risks your team has already agreed to block.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for dependency-review-action

  • GitHub says dependency review shows added, removed, and updated dependencies, plus release dates, dependents, and vulnerability data in pull requests. [1]
  • GitHub says the dependency review action scans vulnerable dependency versions introduced by package changes in pull requests and can fail the check. [1]
  • The GitHub configuration example runs the action on pull_request with contents: read permission. [2]
  • The action README lists fail-on-severity, fail-on-scopes, license controls, package-deny controls, and warn-only as configuration options. [3]
  • GitHub says dependency review and the dependency submission API can race unless dependency submission runs before review or review retries snapshot warnings. [1]

Dependency review works best as a narrow merge gate

Bottom line: Run dependency review on pull requests, keep the job read-only unless comments are required, choose explicit severity and license rules, and audit the surrounding workflow before treating the check as supply-chain coverage.

GitHub describes dependency review as a way to understand dependency changes and their security impact at every pull request. It can show package additions, removals, updates, release dates, dependents, and vulnerability data. [1]

That makes it a good merge gate for package changes, not a replacement for workflow hardening. The action can tell you what dependency changed; it does not prove that the workflow invoking it is pinned, least-privileged, timeout-bounded, or safe around untrusted pull-request input.

The action turns dependency review into a CI check

GitHub says the dependency review action reports on pull-request differences inside GitHub Actions and adds enforcement mechanisms to the workflow. It scans for vulnerable dependency versions introduced by package version changes and warns about the associated security vulnerabilities. [1]

GitHub also says the check fails by default when it discovers vulnerable packages, and that a failed check can block merging when the repository owner requires the dependency review check to pass. [1]

Clean verdict: Treat dependency review as a required status check only after the workflow file, branch protection, and exception process are all reviewable by maintainers.

Configuration should match the risk you intend to block

GitHub documents two configuration methods for the action: inline options in the workflow file or a referenced configuration file. [2] The separate action README says all configuration options are optional. [3]

The default-free posture is dangerous because teams can mistake installation for policy. The README lists fail-on-severity for vulnerability severity thresholds, fail-on-scopes for runtime, development, or unknown scopes, license controls, deny-packages, deny-groups, and warn-only. [3]

Set the check to fail on the dependency changes you would actually stop in review. A critical-only gate may be reasonable during rollout; a mature repository may want moderate-or-higher runtime vulnerabilities, denied package namespaces, and license rules that mirror legal review.

The basic workflow only needs read access to contents

GitHub's configuration example runs the workflow on pull_request, checks out the repository, invokes actions/dependency-review-action@v4, and grants contents: read. [2] The action README's standard installation example uses the same read-only contents permission with actions/dependency-review-action@v5. [3]

Keep that baseline unless the workflow needs more. The README says comment-summary-in-pr can report the review summary as a pull-request comment, and that enabling it requires pull-requests: write. [3]

Avoid this posture: A dependency review workflow grants broad write permissions, runs extra scripts from the pull request, and then treats the passing dependency check as proof that the CI path is safe.

Submitted dependencies need ordering, not hope

GitHub says the action uses the dependency review REST API to compare dependency changes between base and head commits, and that it also considers dependencies submitted through the dependency submission API. [1]

GitHub warns that when dependencies are scanned during a build and uploaded through dependency submission, teams should avoid a race condition between submission and review. For GitHub Actions, it recommends running dependency submission actions in the same workflow as dependency review, or using retry-on-snapshot-warnings with a timeout that slightly exceeds the longest dependency submission run. [1]

That ordering matters for generated lockfiles, vendored manifests, build-system plugins, and language ecosystems where the dependency graph depends on submitted snapshots.

Dependency review does not audit the workflow around it

GitHub says dependency review supports the same languages and package management ecosystems as the dependency graph. [1] That ecosystem boundary is useful context when a repository uses generated dependencies, private package metadata, or build-time dependency submission.

Review the dependency gate beside the workflow controls that protect the gate itself: pinned third-party actions, explicit GITHUB_TOKEN permissions, safe pull-request triggers, bounded runtimes, trusted reusable workflows, and secrets that do not enter untrusted jobs.

A fast dependency-review audit starts with the PR file

Open the workflow that runs dependency review and confirm it uses a pull-request event, a pinned or deliberately versioned action reference, and the narrow permissions documented in the GitHub examples. [2] Then inspect the policy: severity threshold, scopes, license treatment, package or namespace denies, and whether warn-only disables enforcement. [3]

If your repository uses dependency submission, check that submission completes before review or that retry settings cover the expected run time. [1] Finally, require the check only after maintainers understand what it can and cannot block.

Run gha-guard locally so dependency review sits beside pinned-action, token-permission, timeout, OIDC, reusable-workflow, runner, cache, attestation, and shell-context checks.
