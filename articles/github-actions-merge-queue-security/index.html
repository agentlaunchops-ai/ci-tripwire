GitHub Merge Queue: Secure GitHub Actions CI Checks

By CI Tripwire Editorial · Published 2026-06-09 · 7 min read

GitHub Actions merge queue workflows need explicit event coverage so required checks still report on the synthetic merge group.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for GitHub Actions merge queue checks

  • GitHub Docs says a merge queue can require pull request changes to pass required status checks when applied to the latest target branch and earlier queued pull requests. [1]
  • GitHub Docs says GitHub Actions workflows that provide required pull request checks must include the merge_group event when a repository uses a merge queue. [2]
  • GitHub Docs says the supported merge_group activity type is checks_requested. [2]
  • GitHub Docs says a merge queue will wait for required checks to be reported before it proceeds with merging. [1]
  • GitHub Docs says required status checks must have a successful, skipped, or neutral status before collaborators can change a protected branch. [3]

Merge queue checks should run on the queue ref

Bottom line: A workflow that protects normal pull requests may still miss the merge queue unless it listens for merge_group. Audit every required check before enabling the queue on a protected branch.

GitHub Docs says a merge queue checks pull request changes against the latest target branch and any earlier pull requests already in the queue. [1]

That is the point of the feature: it reduces the gap between a pull request passing by itself and the combined queue actually being safe to merge. But it also means CI has to run on the queue's synthetic ref, not only on the original pull request branch.

The merge_group event is separate from pull_request and push

GitHub Docs says the merge_group event is separate from pull_request and push, and warns that required status checks will not trigger when a pull request is added to a merge queue unless the workflow includes merge_group. [2]

The same reference lists the checks_requested activity type for merge_group and shows a workflow that keeps pull_request while adding merge_group with types: [checks_requested]. [2]

That distinction matters in security reviews. A workflow can look mature because it runs on pull requests, uses required checks, and blocks direct branch changes, while still leaving the queue path without the exact check name the protected branch expects.

Clean verdict: Required checks that gate a merge queue should name both the pull request path and the merge_group path in the workflow trigger review.

Required checks turn missing queue coverage into a merge blocker

GitHub Docs says a merge queue will wait for required checks to be reported before it can proceed, and that CI must trigger and report on merge group events when requiring a merge queue. [1]

GitHub Docs also says required status checks must be successful, skipped, or neutral before collaborators can make changes to a protected branch. [3]

Together, those rules make a missing trigger noisy instead of silent. The queue waits for a check result, the workflow never starts on the merge group, and maintainers see a blocked queue instead of the same CI signal they saw on the pull request.

The safer setup is to map every branch-protection check name back to the workflow and job that emits it. GitHub Docs warns that duplicate job names across workflows can cause ambiguous status check results and block pull request merges. [3]

Queue settings can hide a stale CI assumption

GitHub Docs says repository administrators can require a merge queue through branch protection for the base branch. [1]

The merge queue settings include build concurrency, which GitHub Docs describes as the maximum number of merge_group webhooks to dispatch, from 1 to 100. [1]

GitHub Docs also documents an "Only merge non-failing pull requests" setting. When enabled, all pull requests must satisfy required checks to be merged; when disabled, a grouped set can proceed if the last pull request in the group has passed checks for the combined changes. [1]

These are operational controls, but they affect security expectations. A high-concurrency queue can dispatch many merge groups, and an intermittent-test policy can change which failures hold up the queue. Reviewers should understand those choices before treating the queue as a simple branch-protection replacement.

Avoid verdict: Do not enable merge queue on a protected branch until required check names, queue triggers, and queue settings have been reviewed together.

A fast audit starts with every required check name

Start from branch protection or rulesets and list every required status check. GitHub Docs says required status checks are part of branch protection and must reach an accepted status before protected-branch changes can proceed. [3]

Then inspect the workflow that emits each required check. If the branch uses merge queue and the workflow only listens to pull_request or push, add a queue-specific review item because GitHub Docs says merge_group is separate from those events. [2]

Finally, test a queue entry with the actual protected branch. GitHub Docs says merge queue creates temporary branches for validation and waits for CI responses from those merge group branches. [1]
