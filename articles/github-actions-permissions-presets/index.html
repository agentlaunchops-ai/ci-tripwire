GitHub Actions Permissions: Avoid write-all CI Tokens

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

GitHub Actions permissions presets make token scope readable, but read-all, write-all, and {} can also hide whether each job has the narrow access it actually needs.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for GitHub Actions permissions presets

  • GitHub Docs says permissions can modify the default permissions granted to the GITHUB_TOKEN. [1]
  • GitHub Docs says permissions can be used at the workflow level or for individual jobs. [1]
  • GitHub Docs lists read-all, write-all, and {} as supported permission presets. [1]
  • GitHub Docs says unspecified permissions are set to none after you define access for any available permission. [1]
  • GitHub Docs says the github context includes sensitive information such as github.token. [3]

Permissions presets should start narrow, then prove exceptions

Bottom line: Treat write-all as an exception, not a default. Prefer explicit job-level scopes so the job that comments, releases, scans, or deploys is the only job carrying that access.

GitHub Docs says workflows can use permissions to modify the default permissions granted to the GITHUB_TOKEN, and it shows the key at workflow and job level. [1]

That makes permissions review possible in code. The trap is that a compact preset can look tidy while granting every current scope, denying everything, or spreading a broad default across jobs that only need read access.

read-all, write-all, and empty permissions mean very different things

The workflow syntax reference lists read-all to give read access to all available permissions and write-all to give write access to all available permissions. [1]

The same reference says permissions: {} disables permissions for all available scopes. [1]

Those three presets are useful review signals. {} can make a no-token job clear. read-all can be acceptable for broad inspection jobs. write-all should trigger a line-by-line explanation because it grants write access across the available token scopes.

Clean verdict: A job with permissions: contents: read and one extra write scope is easier to defend than a workflow with permissions: write-all at the top.

Workflow-level permissions become every job's starting point

GitHub Docs says permissions can be used either for all jobs in a workflow or within specific jobs. [1]

That placement changes the review question. A workflow-level preset sets the initial token posture for the whole file, while a job-level block explains why a particular job needs a particular scope.

GitHub Docs also says that after specifying access for any available permission, all unspecified permissions are set to none. [1]

That rule is a strong reason to write the scopes out. Explicit job blocks keep future readers from guessing whether a release job really needs package write access, whether a scan job needs security-events write access, or whether a test job needs anything beyond repository read access.

Token scope matters more when jobs handle untrusted data

GitHub's token tutorial says actions can access the GITHUB_TOKEN through the github.token context even when the workflow does not explicitly pass the token to the action. [2]

That implicit availability matters during review. GitHub Docs says the github context includes sensitive information, such as github.token. [3]

GitHub's secure-use reference warns that some workflow contexts contain untrusted input and should be treated carefully in scripts. [4]

Put those points together before approving broad presets. If a job reads pull-request titles, issue text, release notes, dispatch inputs, or branch names into shell commands, its token scope determines how much damage a script-injection mistake can reach.

Avoid verdict: Do not combine untrusted event data, shell execution, and write-all unless the job has a specific reason that survives review.

A fast audit starts by finding every permissions block

Search workflow files for permissions:, read-all, write-all, and permissions: {}. GitHub Docs lists those presets and says the key controls GITHUB_TOKEN access, so each match should have a job-level reason. [1]

Then review whether the block sits at workflow level or job level. Jobs that only build, lint, or test usually deserve lower token access than jobs that publish packages, create releases, upload SARIF, comment on pull requests, or deploy infrastructure.

Finally, pair token scope with trigger and script review. A broad token on a trusted release path is different from a broad token on a workflow that processes untrusted issue, pull-request, or manual-dispatch data.
