GitHub Actions id-token: Secure OIDC Deploy Workflows

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

GitHub Actions OIDC removes long-lived cloud secrets from many deploy jobs, but the id-token: write permission still deserves a tight review before a workflow can request a cloud-facing identity token.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for GitHub Actions id-token security

  • GitHub Docs says OIDC lets workflows request short-lived access tokens directly from cloud providers. [1]
  • GitHub Docs says a job or workflow needs permissions: id-token: write to fetch an OIDC JSON Web Token. [1]
  • GitHub Docs says id-token: write allows OIDC token fetching but does not grant write access to other resources. [1]
  • GitHub Docs says the OIDC token includes claims such as aud, sub, repository, ref, workflow, and environment. [3]
  • GitHub Docs says audience and subject claims are typically used together when cloud roles define trust conditions. [3]

id-token is a minting permission, not a deploy policy

Bottom line: Give id-token: write only to jobs that actually exchange OIDC tokens, then make the cloud role enforce the repository, ref, workflow, environment, audience, and subject you intended.

GitHub Docs says OpenID Connect can let GitHub Actions workflows request short-lived access tokens directly from cloud providers, which can remove the need to store long-lived cloud credentials as GitHub secrets. [1]

That is a better credential shape, but it is not an automatic deploy boundary. The workflow still decides which job can ask GitHub for an identity token, and the cloud provider still decides which token claims are allowed to assume a role.

The id-token permission should sit on the job that needs it

GitHub Docs says a workflow or job requires permissions: id-token: write before it can fetch an OIDC token from GitHub's OIDC provider. [1]

The workflow syntax reference says the id-token permission fetches an OpenID Connect token and requires id-token: write. [2]

GitHub Docs also notes that setting id-token: write does not give the workflow permission to modify or write to resources; it only allows the workflow to request and use an OIDC token. [1]

That detail is useful for review. id-token: write is not the same thing as repository write permission, but it is still the permission that lets a job mint the identity material a cloud provider may trust.

Clean verdict: A deploy job with explicit contents: read and id-token: write is easier to review than a workflow-wide permissions block that gives every job token-minting ability.

Cloud trust should match the OIDC claims the job emits

GitHub's OIDC reference says the token includes standard claims such as aud, iss, and sub, and it identifies the token type as a JSON Web Token. [3]

The same reference lists GitHub-provided claims including repository, ref, workflow, environment, event_name, runner_environment, workflow_ref, and workflow_sha. [3]

GitHub Docs says audience and subject claims are typically used together when cloud roles set trust conditions for GitHub workflows. [3]

For a deploy workflow, those claims are the review surface. If the cloud role accepts any branch, any workflow file, or any environment name, a tidy id-token block in YAML can still point at an overly broad cloud trust policy.

OIDC deploy jobs still need ordinary workflow hardening

GitHub's secure-use reference warns that some workflow contexts contain untrusted input and should be treated carefully in scripts. [4]

That warning still applies when the credential is short-lived. A deploy job can be safer without static cloud keys and still be risky if untrusted event data controls a shell command, release tag, Docker image name, or deploy target.

Keep the OIDC exchange close to the job that needs it. Put build and test jobs on lower permissions, place deploy jobs behind environment protection when the repository uses environments, and avoid passing unvalidated workflow inputs into the same step that exchanges the token.

Avoid verdict: Do not treat OIDC migration as the end of the security review; it changes credential lifetime, but it does not review triggers, scripts, artifact trust, or cloud role conditions for you.

A fast audit starts with every id-token permission block

Search workflow files for id-token, then identify whether the permission is set at workflow level or job level. GitHub Docs says the permission is required for OIDC token fetching, so each occurrence should point to a job that really exchanges a token. [1]

Next, inspect the cloud role trust policy against GitHub's OIDC claims. The OIDC reference lists repository, ref, workflow, environment, audience, and subject data that can be used to constrain which workflow runs receive cloud access. [3]

Then review the deploy job like any other privileged CI path: pinned actions, narrow GITHUB_TOKEN permissions, timeouts, guarded trigger filters, protected environments, and shell handling around event data.
