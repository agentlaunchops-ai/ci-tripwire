GitHub Runner Labels: Safer Self-Hosted CI Routing

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

Self-hosted runner labels should be reviewed as routing policy, because they decide which machines are eligible to execute a GitHub Actions job.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for GitHub runner labels

  • GitHub Docs says self-hosted runners can be selected in a workflow with labels or groups. [1]
  • GitHub Docs says default labels include self-hosted, an operating-system label, and a hardware-architecture label. [1]
  • GitHub Docs says labels are cumulative, so a self-hosted runner must match every label listed in runs-on to be eligible. [1]
  • GitHub Docs says labels are case-insensitive and unused custom labels are automatically deleted within 24 hours. [2]
  • GitHub Docs recommends using self-hosted runners only with private repositories because public-repository forks can run dangerous code on the runner machine through pull requests. [3]

Runner labels decide which machines may execute CI

Bottom line: Treat self-hosted runner labels as a machine-routing control. Use runner groups for repository access, then use labels to narrow jobs to the exact operating system, architecture, and capability they need.

GitHub's workflow guide says self-hosted runners can be selected by labels or by groups. [1]

That makes runner labels more than a convenience. When a job says runs-on: [self-hosted, linux, x64, gpu], the labels decide which runner machines are eligible to receive that job. [1]

Labels are routing selectors, not security badges

GitHub Docs says a self-hosted runner automatically receives labels that indicate it is self-hosted, which operating system it uses, and which hardware architecture it uses. [1]

The same guide says a workflow can combine those labels in runs-on, and that the matching self-hosted runner becomes eligible only when it has all of the listed labels. [1]

That cumulative behavior is useful, but it is not a proof that the machine is safe. It is a selector. A label like gpu, prod-network, or signing should be backed by an inventory review showing which runner owns the capability and why that job needs it.

Runner groups set repository access before labels narrow jobs

GitHub Docs says runner groups can limit which repositories have access to self-hosted runners that were added to an organization. [3]

The same page says a new organization runner group requires a repository-access policy, and that the group can be made available to selected repositories or to all repositories in the organization. [3]

That gives a clean review split. Runner groups decide which repositories can reach a pool. Labels decide which machines inside the reachable pool can take a job. If the group is broad and the label is broad, the workflow is broad even when the YAML looks specific.

Clean verdict: Put sensitive runners in selected-repository groups first, then use labels for capabilities such as operating system, architecture, hardware, network reach, or signing access.

Default labels should still be checked against reality

GitHub Docs lists self-hosted, linux, windows, macOS, x64, ARM, and ARM64 as default labels for self-hosted runner routing. [1]

GitHub's label guide also says the configuration script can assign default labels such as x64 or linux, and that GitHub Actions accepts those labels as given without validating that the runner actually uses that operating system or architecture. [2]

That means default labels deserve the same inventory hygiene as custom labels. If a runner was rebuilt, replaced, or registered manually, the label set should be checked against the actual host before sensitive jobs depend on it.

Custom labels should name capabilities reviewers can verify

GitHub Docs says custom labels can be created and assigned to self-hosted runners at repository and organization levels, and labels can be used to send jobs to particular types of self-hosted runners. [2]

The workflow guide gives a graphics-hardware example: a gpu label can be assigned to runners that have the required hardware, and a job can combine self-hosted, linux, x64, and gpu. [1]

Use that pattern for clarity. Avoid labels that describe ownership or vibes, such as team-a or trusted, unless they map to a concrete access boundary. Better labels name the specific capability the job consumes.

Avoid verdict: Do not let broad labels such as prod or secure become a hidden permission model.

A fast audit compares YAML selectors to runner inventory

Start by listing every workflow job that uses self-hosted in runs-on. For each job, record the complete label list, the runner group if one is named, and the capability the job is expected to use.

Next, compare those selectors with the runner inventory. GitHub Docs says repository users with repo: write access can view available runners for a repository, including their labels. [1]

Then check repository exposure. GitHub Docs warns that public-repository forks can potentially run dangerous code on self-hosted runner machines through pull-request workflows, and recommends self-hosted runners only for private repositories. [3]

Finally, remember the host itself. GitHub Docs says self-hosted runners are systems you deploy and manage, and that you are responsible for maintaining the runner machines and software beyond automatic updates to the runner application. [4]
