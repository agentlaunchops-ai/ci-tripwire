GitHub Actions Fork Approvals: Safer PR CI Workflows

By CI Tripwire Editorial · Published 2026-06-08 · 7 min read

Fork pull requests are useful collaboration paths, but they can also ask your CI to run code from a branch your maintainers do not control.

This article is informational and is not a substitute for a security review of your own workflows, repositories, or cloud accounts.

Key takeaways for fork workflow approvals

  • GitHub says workflow runs triggered by contributor pull requests from forks may require approval from a maintainer with write access. [1]
  • GitHub says approval requirements can be configured at repository, organization, or enterprise level. [1]
  • GitHub's repository settings include a mode that requires approval for fork pull-request workflows from collaborators without write permission. [2]
  • GitHub says waiting workflow runs are automatically deleted after more than 30 days. [1]
  • GitHub Docs state that forked-repository pull requests receive a read-only GITHUB_TOKEN. [3]

Fork approval is a CI trust decision

Bottom line: Treat a fork-workflow approval as permission to run untrusted pull-request code through your CI. Review workflow-file changes, token posture, and secrets exposure before clicking approve.

GitHub's approval documentation says maintainers with write access can approve workflow runs triggered by a contributor's pull request from a public fork. [1]

That approval is not just queue management. It decides whether the repository's workflow definitions, runners, caches, artifacts, and job permissions should process code that came from outside the trusted branch set.

Approval gates should slow down risky first runs

GitHub states that fork pull-request workflow runs may require manual approval from a maintainer with write access, and that approval requirements can be configured for a repository, organization, or enterprise. [1]

GitHub also says runs awaiting approval for more than 30 days are automatically deleted. [1]

Clean verdict: A queued fork run is a review checkpoint, not a nuisance. If nobody can explain why the run is safe, letting it expire is a valid outcome.

The practical review question is simple: would this workflow still be acceptable if every checked-out file, test fixture, build script, and package hook came from an untrusted contributor branch?

Repository settings decide who gets the prompt

GitHub's repository settings documentation says anyone can fork a public repository and submit a pull request that proposes changes to the repository's GitHub Actions workflows. [2]

The same settings page documents fork pull-request workflow approval choices, including approval for first-time contributors, first-time contributors who are new to GitHub, and collaborators without write permission. [2]

For public projects with meaningful release automation, the collaborator-without-write-permission setting is easier to reason about than a narrow first-time-only gate, because prior contribution history does not automatically mean the next CI run is safe.

Review workflow-file changes before approval

GitHub's approval steps tell maintainers to inspect the pull request's changed files and be especially alert to proposed changes under .github/workflows/. [1]

That warning deserves a concrete checklist. Look for new triggers, broader permissions, new pull_request_target paths, shell steps that interpolate pull-request text, third-party actions that are not pinned, new artifact uploads, and jobs that reach deployment environments.

Avoid this posture: A maintainer approves a fork run after reading only the application diff, while the workflow diff quietly changes tokens, scripts, or release-adjacent jobs.

Read-only fork tokens are not a complete boundary

GitHub Docs say the GITHUB_TOKEN has read-only permissions in pull requests from forked repositories. [3]

Read-only token posture reduces one common write path, but it does not make every job harmless. CI can still execute build scripts, run package lifecycle hooks, upload logs and artifacts, consume caches, and exercise networked test paths.

If a workflow needs secrets, cloud credentials, write permissions, or deployment authority, keep that work outside untrusted fork execution and behind a trusted event or a separate maintainer-controlled promotion path.

The API can approve a fork run deliberately

GitHub's REST API documentation includes an endpoint to approve a workflow run for a fork pull request from a public fork. [4]

That endpoint is useful for internal tooling only when the tool preserves the same review standard as the web UI. An automated approval path should record the pull request, workflow file changes, run identifier, approver authority, and reason for approval.

Do not turn API approval into a rubber stamp. If the approval decision is automated, the policy should be narrow enough that a maintainer can audit why one fork run passed and another stayed queued.

A fast fork-approval audit starts with settings and workflow diffs

First, review repository and organization Actions settings for fork pull-request approval requirements. Then inspect recent fork pull requests that changed .github/workflows/, broadened tokens, added unpinned actions, or introduced jobs that handle artifacts, caches, secrets, or deployments.

Next, check whether maintainers have a written approval checklist. The useful checklist is short: workflow diff, token permissions, secret access, runner exposure, cache and artifact handling, third-party actions, shell interpolation, and deployment reach.

Run gha-guard locally so fork approval review sits beside pinned-action, token-permission, timeout, pull-request trigger, OIDC, reusable-workflow, runner, cache, attestation, environment, dependency-review, shell-context, required-check, script-injection, allowlist, and artifact-retention audits.
